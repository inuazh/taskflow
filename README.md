## queryclient 

файл queryclient.ts дефолты выбраны под профиль приложения: задачи меняются часто справочники редко

`staleTime 30s` - компромисс. свежие данные 30 секунд после получения. для большинства запросов норм. для справочника пользователей переопределяется в самом хуке `useusers` на 5 минут - список людей не меняется каждые полминуты
`gcTime: 5min` - дефолт tanstack query. неиспользуемые записи живут в памяти 5 минут после размонтирования последнего подписчика
`retry: 2` - при сетевой ошибке до 2 повторов. итого максимум 3 сетевых вызова при сбое
`refetchOnWindowFocus: true` - рефетч при возврате во вкладку но только для stale-данных. это закрывает требование 6.3 тз
`mutations.retry: 0` - мутации не повторяются автоматически чтобы не плодить дубли (например две созданных задачи). повтор явно через кнопку в тосте

## разграничение

### tanstack query - серверные данные

всё что приходит с сервера. список задач список юзеров. кеш по query key - `tasks/list/{params}` где params это `{page limit q}`. каждая комбинация это отдельная запись кеша. это даёт автоматический prefetch и мгновенное переключение между уже посещёнными страницами

### url - фильтры пагинация поиск

то что должно шериться по ссылке и работать с кнопкой "назад" в браузере. через tanstack router с zod-схемой

выбор url > zustand для этих параметров намеренный. тз говорит фильтры в zustand но рядом -"поисковый запрос сохраняется в url" не делаю двустороннюю синхронизацию (стор to url) вместо этого url первичен и единственная правда. компонент читает через `usesearch()` и передаёт в `usetasks`- это становится частью query key

### zustand - чисто клиентский ui

то что не должно быть в url и не относится к серверу

`tasksuistore`:
 `editingid: number | null` - какая задача сейчас в режиме редактирования (граничный случай со стр. 6 тз - модалка не должна быть в url)
 `selectedids: number[]` - массовый выбор для bulk-операций. сбрасывается после операции

`globaluistore` (с persist в localStorage):
 `density: 'compact' | 'comfortable'` - плотность списка. переживает перезагрузку

компоненты подписываются через селекторы `usetasksuistore(s => s.editingid)`. подписка точечная - изменение `selectedids` не перерендерит компонент который читает только `editingid`

## оптимистичные обновления

реализованы для всех мутаций - create update updatestatus delete. паттерн одинаков:

1 `onmutate` - `cancelqueries` отменяет летящие фоновые запросы за `taskkeys.all` чтобы их ответ не затёр оптимистичное изменение. потом snapshot текущего кеша через `getqueriesdata`. потом ручное `setqueriesdata` с обновлёнными данными. snapshot возвращается в context
2 `onerror` - откат: проходим по сохранённым парам `[key data]` и кладём обратно через `setquerydata`. показываем тост
3 `onsettled` - `invalidatequeries` для финальной синхронизации с сервером (на dummyjson нивелирует оптимистику но на реальном api это критично)

## граничные случаи (с. 6 тз)

тз явно просит обосновать выбор инструмента в неочевидных ситуациях

**фильтры в url и одновременно в сторе - как синхронизировать?**
не синхронизируем. url первичен и единственнаяправда. синхронизация двух источников приводит к рассинхрону. zustand для фильтров не используется - вместо этого читает из url через usesearch

**после удаления задачи нужно перейти на предыдущую страницу если текущая опустела - кто управляет этой логикой?**
компонент `tasklist`. хук `usedeletetask` ничего не знает про роутер и про текущую страницу - он умеет только удалять задачу из кеша. компонент через `useeffect` смотрит на `data.todos.length` и `page` и при необходимости вызывает `navigate`   - мутация переиспользуема (например в массовом удалении) и не тащит за собой ui-логику

**форма редактирования: где живут грязные данные?**

в rhf. `defaultvalues` берутся из объекта task (который пришёл из кеша query). дальше форма живёт сама, даже если в фоне обновится кеш query пользователь не теряет то что напечатал. при сабмите берём данные из rhf и шлём в мутацию. это и есть три уровня: query (сервер) rhf (форма) zustand (открыта/закрыта)

**поиск с дебаунсом - одно состояние или два?**
два. локальный `usestate` для значения инпута (мгновенный отклик), `usedebounce(value 300)` для отложенного значения, `useeffect` пушит дебаунс в url. инпут живёт в react

## ограничения и компромиссы

 **dummyjson не хранит мутации.** все create/update/delete возвращают success но при следующем рефетче данные пропадают. оптимистика реализована - на реальном api всё работало бы. видно в момент мутации задача появляется/изменяется/исчезает мгновенно потом откатывается из-за рефетча в `onsettled`

 **поиск только на клиенте.** dummyjson не предоставляет search-эндпоинт для `/todos`. фильтрация по тексту происходит на клиенте на загруженной странице - это не полнотекстовый поиск по всей базе. в реальном api заменилось бы на серверный поиск без изменения архитектуры (только `gettasks` стал бы передавать `q` в url)

 **фильтры по приоритету и тегам не реализованы.** dummyjson не предоставляет таких полей. сделал фильтры которые поддерживает api: пагинация и текстовый поиск

## не успел

- online/offline баннер с автоматическим рефетчем (раздел 6.3)
- refetchinterval для эмуляции изменений другим пользователем (6.3)
- режимы канбан/таблица - есть только список (плотность переключается)
